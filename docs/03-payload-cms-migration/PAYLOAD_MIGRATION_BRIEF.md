# Payload CMS Migration Brief

## Objective

Migrate the portfolio website from its current content structure to a Payload-powered CMS after the core positioning and content rewrite are complete.

The goal is to make the portfolio easier to manage, extend, preview, and use as a product design platform.

## Why Payload

Payload should support:

- Structured case studies
- Draft case studies
- Lab experiments
- Writing
- Visual archive
- Video content
- Project metadata
- Reusable taxonomy
- Admin editing
- Preview workflows
- Versioning
- Future product/project workflows

Payload supports drafts for collections and globals, allowing unpublished versions to be saved before publishing. This is useful for case studies, writing, and project pages that need editing before launch.

## Migration Timing

Do not start this phase until:

- Homepage positioning is updated.
- Case study structure is defined.
- Figma/Payload case study placeholder exists.
- Core content model is known.
- Current site IA has been tested.

## Migration Goals

### P0

- Create Payload project or integrate Payload into the site architecture.
- Define content collections.
- Migrate case studies.
- Migrate writing.
- Migrate projects/lab entries.
- Add media handling.
- Add basic preview workflow.
- Preserve current routes where possible.

### P1

- Add drafts/versioning.
- Add visual archive collection.
- Add video content collection.
- Add project status fields.
- Add tags/categories.
- Add SEO fields.
- Add Open Graph metadata.

### P2

- Add advanced relationships.
- Add changelogs.
- Add content series.
- Add admin customizations.
- Add richer preview workflows.
- Add private/internal project notes if useful.

## Collections Needed

- Pages
- Case Studies
- Lab Projects
- Writing
- Visual Archive Items
- Videos
- Projects
- Tags
- Tools
- Media
- Testimonials or Quotes, optional

## Globals Needed

- Site Settings
- Navigation
- Homepage
- About Summary
- SEO Defaults
- Social Links

## Important Rule

Do not design the Payload CMS around every possible future idea.

Design it around the content that needs to exist now, then leave room for future expansion.
