# Easy Site Changes Before Payload

## Goal

Make high-impact content and positioning changes before rebuilding the backend with Payload.

## Rules

- Do not migrate to Payload in this phase.
- Do not rewrite the entire app.
- Do not create complex new data systems.
- Do not block content changes on future CMS decisions.
- Use the current site structure where possible.
- Add placeholders where future case studies will go.

## P0 Changes

### 1. Homepage Identity

Change the primary identity from broad designer/developer language to product design language.

Use:

Product Designer for systems, prototypes, and design-dev collaboration.

### 2. Homepage Subheadline

Use:

I design product experiences and build realistic prototypes that help designers, developers, and stakeholders move from ambiguity to buildable solutions.

### 3. Navigation

Prefer:

- Case Studies
- Lab
- Writing
- Visual Archive
- About
- Resume

### 4. Featured Work

Reorder featured work so product design case studies appear before visual/archive work.

### 5. Figma/Payload Placeholder

Add a placeholder case study card for the upcoming flagship project.

Status:

In progress

### 6. About Preview

Rewrite around product design, systems, prototyping, and design-dev collaboration.

### 7. Project Descriptions

Reframe projects around:

- Problem
- Users/stakeholders
- System/workflow
- Design decisions
- Technical constraints
- Outcome

## Definition Of Done

This phase is done when:

- The homepage clearly says Product Designer.
- The first featured project is the Figma/Payload case study placeholder.
- The site has a case-study-first structure.
- Existing work is reframed around product design value.
- The backend has not been significantly refactored.
